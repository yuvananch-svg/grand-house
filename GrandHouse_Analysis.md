## 1. สรุปภาพรวมที่ถูกต้อง

Grand House ไม่ใช่ร้านที่มี POS รายสินค้า แต่เป็นร้านอาหาร/ของพร้อมขายแบบบ้านๆ ที่ขายหน้าร้านโดยรับเงินผ่านหลายช่องทาง และใช้การนับสต็อกย้อนกลับเพื่อประเมินว่าสินค้าแต่ละรายการขายไปกี่กล่อง

ดังนั้นระบบ Google Sheets ไม่ควรถูกออกแบบเหมือนระบบ POS เต็มรูปแบบ แต่ควรเป็นระบบ 3 ชั้น:

1. **Cash Layer** — บันทึกยอดเงินจริงที่รับได้ทั้งวัน
2. **Stock Layer** — บันทึกการเบิก, เหลือ, แถม, เสีย เพื่อคำนวณยอดขายรายสินค้าแบบย้อนกลับ
3. **Reconciliation Layer** — เทียบยอดเงินจริงกับยอดขายที่คำนวณจากสต็อก

ประโยคสำคัญ:

> `บันทึกยอดขาย` ไม่ใช่ชีทซ้ำซ้อนกับ `สต็อกสินค้า`  
> `บันทึกยอดขาย` คือชั้นตรวจยอดเงินสด/QR  
> `สต็อกสินค้า` คือชั้นประเมินยอดขายรายสินค้า  
> สิ่งที่ต้องมีเพิ่มคือการตรวจส่วนต่างระหว่างสองชั้นนี้

## 3. Workflow การขายจริงของร้าน

### 3.1 ระหว่างวัน

Grand House ไม่มี POS ที่บันทึกว่าสินค้าแต่ละรายการขายไปกี่กล่อง ลูกค้าหยิบหรือพนักงานหยิบสินค้าใส่ถุง แล้วรับเงินผ่านช่องทางต่างๆ

ช่องทางรับเงินหลัก:

- QR 1
- QR 2
- เงินสด
- ออนไลน์
- อื่นๆ

ระหว่างวันอาจมีการจดบนกระดาษสำหรับรายการพิเศษ เช่น:

- แถมสินค้า
- ขายราคาโปร
- ขายราคาพิเศษ
- ของเสียบางกรณี
- หมายเหตุอื่นๆ ที่ต้องนำมากรอกตอนเย็น

### 3.2 ตอนปิดวัน

ตอนเย็นหรือช่วงที่สะดวก พนักงานจะทำ 2 อย่าง:

1. รวมยอดเงินจริงจาก QR / เงินสด / ออนไลน์ / อื่นๆ แล้วกรอกใน `บันทึกยอดขาย`
2. นับของเหลือหน้าร้านแยกตามสินค้า แล้วกรอกใน `สต็อกสินค้า`

ตัวอย่างการนับของเหลือ:

| สินค้า | เหลือ |
|---|---:|
| ข้าวผัด | 2 |
| เค้ก | 5 |
| น้ำ | 10 |

จากนั้นระบบควรคำนวณยอดขายรายสินค้าแบบย้อนกลับ:

```text
จำนวนขายได้ = จำนวนเบิก - จำนวนเหลือ - จำนวนแถม - จำนวนเสีย
```

ถ้ามีการนับหลายครั้งในวันเดียวหรือข้ามวัน สูตรต้องเปลี่ยนเป็นการเทียบกับยอดเหลือครั้งก่อน:

```text
ขายใน session นี้ = เหลือครั้งก่อน - เหลือครั้งนี้ - แถมใน session นี้ - เสียใน session นี้
```

### 3.3 การเทียบยอด

หลังจากได้ยอดเงินจริงและยอดขายจากสต็อก ระบบควรเทียบ:

```text
ส่วนต่าง = ยอดเงินจริงจากช่องทางรับเงิน - ยอดขายที่คำนวณจากสต็อก
```

ถ้าส่วนต่างใกล้ 0 แปลว่าการนับสต็อกและเงินรับตรงกัน

ถ้าส่วนต่างมาก แปลว่าอาจมีปัญหา เช่น:

- นับของเหลือผิด
- จดแถมตก
- จดราคาโปร/ราคาพิเศษไม่ครบ
- เงินสดขาดหรือเกิน
- QR ลงผิดวัน
- มีการขายข้ามวันแต่ระบบปัจจุบันตัดรอบผิด

```spreadsheet
=IFERROR(
  INDEX('ต้นทุนข้าวกล่อง'!G$7:G$100,
    MATCH(1,
      ('ต้นทุนข้าวกล่อง'!C$7:C$100=E9) *
        ('ต้นทุนข้าวกล่อง'!I$7:I$100<=E7) *
        (('ต้นทุนข้าวกล่อง'!J$7:J$100>=E7) +
         ('ต้นทุนข้าวกล่อง'!J$7:J$100="")),
    0)
  ),
  "ไม่พบราคา"
)
```

และสูตรต้นทุนต่อกล่อง:

```spreadsheet
=IFERROR(
```

### 5.6 `บันทึกสูตรอาหาร`

สถานะปัจจุบัน:

- มีหัวตารางคร่าวๆ
- ยังไม่มีข้อมูลจริง
- ไม่มีสูตรเชื่อมโยงกับวัตถุดิบ
- ยังไม่สามารถคำนวณต้นทุนเมนูสดต่อกล่องได้

ผลกระทบ:

- เมนูสดยังรู้ต้นทุนจริงไม่ได้
- ถ้ามีเมนูที่ทำสดหน้าร้านและขายในระบบเดียวกัน Dashboard จะไม่รู้ margin ที่ถูกต้อง

### 5.7 `ฐานข้อมูล`

บทบาทปัจจุบัน:

- ดึงข้อมูลจาก `สต็อกสินค้า`
- ดึงข้อมูลจาก `บันทึกค่าใช้จ่าย`
- ทำ aggregation สำหรับ Dashboard
- มีสูตร `FILTER`, `QUERY`, `SORT`, `UNIQUE`, `VSTACK`, `SUMIFS`, `SUMPRODUCT`

ข้อสังเกตสำคัญ:

- `ฐานข้อมูล` ตอนนี้ไม่ใช่ raw database บริสุทธิ์
- บางสูตรผูกกับ filter ของ `แดชบอร์ด` เช่น วันที่เริ่ม/สิ้นสุดและสาขา
- แปลว่าข้อมูลใน layer นี้บางส่วนเปลี่ยนตามตัวเลือกบน Dashboard

ความเสี่ยง:

- ถ้า Dashboard เปลี่ยน filter แล้วฐานข้อมูลเปลี่ยนตาม จะทำให้ตรวจสอบย้อนหลังยาก
- ชื่อ `ฐานข้อมูล` อาจทำให้เข้าใจผิดว่าเป็น raw data ทั้งที่จริงเป็น calculated/filtered layer

ข้อเสนอ:

- แยกเป็น 2 ชั้น:
  - `RawData` หรือ `Transaction Log` = ข้อมูลดิบ ไม่เปลี่ยนตาม Dashboard
  - `Dashboard Summary` = ตารางสรุปที่เปลี่ยนตาม filter ได้

## 7. ปัญหาหลักที่ยืนยันแล้ว

### ปัญหา 1: ไม่มี POS รายสินค้า จึงต้อง reconcile เงินกับสต็อก

นี่ไม่ใช่ข้อผิดพลาดของร้าน แต่เป็นข้อจำกัดจริงของ workflow

สิ่งที่ระบบต้องทำ:

```text
ยอดเงินจริงจากช่องทางรับเงิน
เทียบกับ
ยอดขายที่คำนวณจากการเบิก - เหลือ - แถม - เสีย
```

สิ่งที่ระบบยังขาด:

- หน้าหรือส่วนสรุป `ตรวจยอดปิดวัน`
- ตัวเลขส่วนต่างเงิน
- สถานะว่าปิดยอดผ่านหรือควรตรวจสอบ
- เหตุผลของส่วนต่าง

### ปัญหา 2: ระบบบันทึกสต็อกยังไม่รองรับหลาย session

การทำงานจริง:

- บางทีระหว่างวันร้านว่าง พนักงานอาจอยากนับและบันทึกสต็อกช่วงเช้า
- ต่อมาบ่ายหรือเย็นอาจอยากบันทึกต่อ
- บางสินค้าขายข้ามวันได้
- สินค้าบางตัวอาจเสียหลังค้างหลายวันหรือเพราะอากาศร้อน

ระบบปัจจุบัน:

- บันทึกได้เหมือนเป็นรอบเดียว
- ถ้ากดบันทึกใหม่ script อาจสร้าง LOT ใหม่
- ไม่มี concept ของ session ที่อัปเดต LOT เดิม

ผลกระทบ:

- ยอดเบิกอาจถูกนับซ้ำ
- ข้อมูลคงเหลือผิด
- ยอดขายคำนวณผิด
- ตรวจสอบย้อนหลังไม่ได้ว่า session ไหนนับผิด

### ปัญหา 3: ต้องแยก LOT กับ Session

นิยามใหม่ที่ควรใช้:

| คำ | ความหมาย |
|---|---|
| LOT | สินค้าก้อนที่เบิกเข้ามา เช่น ข้าวผัด 10 กล่อง |
| Session | รอบการนับ/อัปเดต เช่น เช้า, บ่าย, เย็น, วันถัดไป |

ตัวอย่าง:

| LOT | Session | วันที่/เวลา | สินค้า | เหลือ | แถม | เสีย | หมายเหตุ |
|---|---|---|---|---:|---:|---:|---|
| LOT-001 | S1 | 23 พ.ค. เช้า | ข้าวผัด | 7 | 0 | 0 | นับระหว่างวัน |
| LOT-001 | S2 | 23 พ.ค. เย็น | ข้าวผัด | 2 | 1 | 0 | ปิดวัน |
| LOT-001 | S3 | 24 พ.ค. เย็น | ข้าวผัด | 0 | 0 | 1 | ข้ามวันแล้วเสีย |

สูตร:

```text
ขาย session แรก = จำนวนเบิก - เหลือครั้งนี้ - แถม - เสีย
ขาย session ถัดไป = เหลือครั้งก่อน - เหลือครั้งนี้ - แถม - เสีย
```

### ปัญหา 4: ระบบแถมต้องระบุสินค้าและจำนวน

Tung ยืนยันว่าเวลาจดแถม จะจดชื่อสินค้าพร้อมจำนวน

ดังนั้นระบบควรเก็บ:

- วันที่
- สาขา

ตัวอย่าง:

### ปัญหา 5: ของเสียไม่ได้เกิดทุกวัน แต่เกิดเมื่อสินค้าเสียจริง

Tung ยืนยันว่า:

- ของเสียจะจดเมื่อของเหลือหลายวันแล้วขายไม่หมด
- หรือบางครั้งอาหารเสียเพราะอากาศร้อน
- ไม่ใช่ทุกของที่ขายไม่หมดจะเป็นของเสียทันที

ดังนั้นต้องแยก:

| ประเภท | ความหมาย |
|---|---|
| เหลือ | ยังขายต่อได้ หรือรอขายใน session/วันถัดไป |
| เสีย | ตัดออกจากสต็อกถาวร เป็น loss |
| แถม | ออกจากสต็อกโดยไม่มีรายได้ |
| ขายโปร/พนักงาน | ออกจากสต็อกโดยมีรายได้ต่ำกว่าราคาปกติ |

### ปัญหา 6: "ราคาพิเศษ" คือเครื่องมือระบายของ ครอบคลุมทุกกรณีลดราคา

- ลดให้ลูกค้าประจำ

ข้อมูลนี้จะทำให้วิเคราะห์ได้ภายหลังว่าเงินหายเพราะ discount แบบไหน

- ให้เลือก LOT_ID ที่ยังเปิดอยู่

- ผู้ใช้กรอกเหลือครั้งนี้, แถม, เสีย, ราคาโปร
- ระบบ append session ใหม่

### 8.5 เพิ่มชีท `คลังสินค้า` — Current State View

ชีทนี้มีไว้ให้พนักงาน**ดูด้วยตาเฉยๆ** ว่าแต่ละ LOT เหลืออะไรอยู่กี่กล่อง — ไม่ต้องกรอกข้อมูลเองเลย

**กฎเหล็ก: ชีทนี้เป็น formula อย่างเดียว ห้ามกรอกมือ**

ถ้าพนักงานต้องอัปเดตทั้งสองชีทเอง จะเกิดสองแหล่งความจริงและตัวเลขจะแตกกัน ชีทนี้ต้องดึงข้อมูลจาก `สต็อกสินค้า` โดยอัตโนมัติ

**โครงสร้าง:**

| คอลัมน์ | ที่มา |
|---|---|
| LOT | ดึงจาก session log |
| สินค้า | ดึงจาก session log |
| วันที่เบิก | ดึงจาก LOT แรกของ LOT นั้น |
| เบิกมา | ดึงจาก session แรก (base qty) |
| คงเหลือ | เหลือครั้งนี้ของ session ล่าสุดของแต่ละ LOT |
| สถานะ | เปิดอยู่ / ขายหมด / ปิดเพราะเสีย |

**กฎการแสดงผล:**

- แสดง **ทีละ LOT แยกกัน** เสมอ — ห้ามรวมยอดข้ามสองสต็อกเข้าหากัน
- ตัวอย่าง: LOT-001 ข้าวผัดเหลือ 3 กล่อง และ LOT-005 ข้าวผัดเหลือ 5 กล่อง → แสดงแยก 2 แถว ไม่ใช่รวมเป็น 8 กล่อง
- เหตุผล: LOT เก่ากว่าควรขายก่อน ถ้ารวมกันจะไม่รู้ว่าต้องรีบขาย LOT ไหน
- Filter default: แสดงเฉพาะ LOT ที่สถานะ "เปิดอยู่" เพื่อไม่ให้รก

**เวลาที่ควรสร้าง:** หลัง session log ใน Phase 2 นิ่งแล้ว — สูตรต้องอาศัย session log ที่ทำงานได้ก่อน

## 9. Flow ระบบใหม่ที่ตรงกับการทำงานจริง

```text
1. รับสินค้า/เบิกสินค้า
   -> สร้าง LOT ใหม่ใน Stock Lots

2. ระหว่างวันขายสินค้า
   -> รับเงินผ่าน QR1 / QR2 / เงินสด / ออนไลน์ / อื่นๆ
   -> จดแถม/โปร/พนักงานบนกระดาษถ้ามี

3. ระหว่างวันถ้าว่าง
   -> นับสต็อกบางรายการ
   -> อัปเดต LOT เดิมผ่าน Stock Sessions

4. ปิดวัน
   -> กรอกยอดเงินจริงใน บันทึกยอดขาย
   -> นับของเหลือแยกสินค้า
   -> กรอก session ปิดวัน

5. ระบบคำนวณ
   -> ขายได้แต่ละสินค้า
   -> รายได้คำนวณจากสต็อก
   -> ต้นทุนออกจากสต็อก
   -> แถม
   -> เสีย
   -> คงเหลือล่าสุด

6. ตรวจยอด
   -> เทียบยอดเงินจริงกับยอดขายจากสต็อก
   -> แสดงส่วนต่างและสถานะ

7. Dashboard / P&L
   -> ดึงจาก raw logs และ summary ที่ผ่าน logic แล้ว
```

## 10. สิ่งที่ควรแก้ในไฟล์เดิม

### แก้ความเข้าใจเรื่อง `บันทึกยอดขาย`

ข้อมูลเดิมอาจทำให้เข้าใจว่า `บันทึกยอดขาย` เป็น sales source ที่ควรบอกยอดขายทั้งหมด แต่ความจริงคือ:

```text
บันทึกยอดขาย = เงินสด/QR/ช่องทางรับเงิน
สต็อกสินค้า = ยอดขายรายสินค้าแบบคำนวณย้อนกลับ
```

ต้องใช้คู่กัน ไม่ใช่เลือกอย่างใดอย่างหนึ่ง

### แก้ข้อเสนอเรื่องการบวกต้นทุนแถม

ข้อเสนอเดิมที่ว่า:

```text
ต้นทุนรวม = จำนวนเบิก x ต้นทุนต่อกล่อง + ต้นทุนของแถม
```

อาจผิดได้ เพราะถ้าแถมเป็นสินค้าจาก LOT เดียวกัน การคิดจำนวนเบิกทั้งหมดเป็นต้นทุนอยู่แล้ว การบวกต้นทุนแถมซ้ำอาจทำให้ double count

วิธีที่ถูกกว่า:

- ใช้ movement/session log
- ต้นทุนควรเกิดจากการเคลื่อนไหวออกจากสต็อก:
  - ขาย
  - แถม
  - เสีย
- ไม่ควรบวกแถมแบบแยกโดยไม่รู้ว่าแถมมาจาก LOT ไหน

### แก้เรื่องของเสีย

ของเสียไม่ใช่สิ่งที่เกิดจากยอดไม่ขายได้ในวันนั้นทั้งหมด เพราะบางสินค้าเหลือแล้วขายต่อได้

ต้องแยก:

```text
เหลือ = ยังอยู่ในสต็อก
เสีย = ตัดออกจากสต็อกและรับรู้ loss
```

### แก้เรื่อง Session

ระบบเดิมยังขาดการบันทึกหลายรอบต่อ LOT

ต้องเพิ่ม:

- session id
- เลือก LOT เดิม
- เหลือครั้งก่อน
- เหลือครั้งนี้
- ขายระหว่าง session
- append-only history

### ✅ Phase 1 — ตรวจยอดปิดวัน (เสร็จแล้ว)

สิ่งที่ทำจริง:

1. แก้สูตร Dashboard packaging `AC`→`AG` (ผู้ใช้แก้เองแล้ว)
2. สร้างชีท `ตรวจยอดปิดวัน` วางหลัง `บันทึกยอดขาย` — เทียบ "เงินรับจริง" vs "รายได้จากสต็อก" รายวัน + สรุปรวม + สถานะ ✓/⚠️ + filter ช่วงวันที่/สาขา
3. แก้ปัญหา **locale**: สูตรรองรับวันที่ที่เป็นข้อความ (text date) ด้วย `DATEVALUE/REGEXEXTRACT` + รวมวันที่จากทั้ง 2 ชีท (union)
4. Threshold ส่วนต่างเก็บใน `ตั้งค่า` column I/J (ค่าเริ่มต้น 20 บาท)

   - ราคาขายและต้นทุนต่อกล่องต้องดึงตามวันที่บันทึก ไม่ใช่ค่าล่าสุดเสมอไป
   - ใช้ INDEX/MATCH + ARRAYFORMULA สามเงื่อนไข: ชื่อสินค้าตรง + วันที่เริ่มต้นทุน ≤ วันที่บันทึก + (วันสิ้นสุดต้นทุน ≥ วันที่บันทึก หรือ วันสิ้นสุดว่าง)
   - สูตรจริงที่ใช้ (ยืนยันจาก Tung — ต้องคง logic นี้ไว้เมื่อ refactor):

   **ราคาขาย (col G ของ `ต้นทุนข้าวกล่อง`):**
   ```spreadsheet
   =IFERROR(
     INDEX('ต้นทุนข้าวกล่อง'!G$7:G$100,
       MATCH(1,
         ARRAYFORMULA(
           ('ต้นทุนข้าวกล่อง'!C$7:C$100=E9) *
           ('ต้นทุนข้าวกล่อง'!I$7:I$100<=E7) *
           (('ต้นทุนข้าวกล่อง'!J$7:J$100>=E7) +
            ('ต้นทุนข้าวกล่อง'!J$7:J$100=""))
         ),
       0)
     ),
   "ไม่พบราคา")
   ```

   **ต้นทุน (col H ของ `ต้นทุนข้าวกล่อง`):**
   ```spreadsheet
   =IFERROR(
     INDEX('ต้นทุนข้าวกล่อง'!H$7:H$100,
       MATCH(1,
         ARRAYFORMULA(
           ('ต้นทุนข้าวกล่อง'!C$7:C$100=E9) *
           ('ต้นทุนข้าวกล่อง'!I$7:I$100<=E7) *
           (('ต้นทุนข้าวกล่อง'!J$7:J$100>=E7) +
            ('ต้นทุนข้าวกล่อง'!J$7:J$100=""))
         ),
       0)
     ),
   "ไม่พบต้นทุน")
   ```

   - `E9` = ชื่อสินค้าที่เลือกในฟอร์ม · `E7` = วันที่บันทึก
   - `col I` = วันที่เริ่มต้นทุน · `col J` = วันสิ้นสุดต้นทุน (ว่าง = ยังใช้อยู่)
   - ARRAYFORMULA จำเป็นสำหรับ Google Sheets — ขาดไม่ได้ มิเช่นนั้น MATCH จะไม่คืนผลถูกต้อง
   - ราคาที่ดึงมา = ราคา ณ วันที่เบิก (LOT สร้างใหม่) หรือวันที่นับ (session อัปเดต)
   - ถ้าสินค้าไม่มีราคาในช่วงวันนั้น → แสดง "ไม่พบราคา" / "ไม่พบต้นทุน" และ script ต้อง block การบันทึก

- ปรับ lookup ราคา/ต้นทุนให้ดึงจาก `ต้นทุนข้าวกล่อง` ช่วง `A7:L2000` เพราะชีทจริงมีโครงสร้างข้อมูลยาวกว่า 100 แถว
- แก้ปัญหาช่วง merge ใน `ต้นทุนข้าวกล่อง` โดยไม่อ่าน range แบบที่ชน merged cells โดยตรง
- แก้ error `SpreadsheetApp.getUi()` ใน context ที่เรียกจาก Apps Script editor ด้วยแนวคิด safe alert / logger
- ทดลอง compact layout เพื่อลดช่องว่าง vertical whitespace ในฟอร์ม

สถานะล่าสุดของสคริปต์:

- เวอร์ชันที่ rollback กลับมาใช้เป็นฐานตอนนี้คือเวอร์ชัน compact ก่อนทดลองย้าย storage ลงแถว 35
- raw log ยังอยู่แถว `200+`
- giveaway log ยังอยู่แถว `900+`
- Session Log ยังเป็น read model ด้านล่างของฟอร์ม
- ยังไม่ถือว่า final เพราะ layout มีโอกาสต้องจัดใหม่อีกนิด และต้องตัดสินใจให้ชัดว่า raw log ควรอยู่ตำแหน่งไหนเพื่อไม่ชนพื้นที่ใช้งานจริง

สิ่งที่ทดลองแล้วแต่ยังไม่ควรใช้เป็นฐาน:

- ทดลองย้าย raw log/storage มาใช้แถว `35:2000` และซ่อนแถวล่าง เพื่อให้พื้นที่ด้านล่างเป็น backend storage
- ผลลัพธ์ทำให้ layout เริ่มปนกันระหว่าง UI กับ storage และทำให้หน้าชีทดูเละกว่าที่ควร
- สรุป: แนวคิด "กันพื้นที่ storage แถว 35:2000" อาจใช้ได้ แต่ต้องออกแบบใหม่อย่างระวัง ไม่ควร patch ต่อจาก layout ปัจจุบันแบบเร็วๆ

งานที่ยังค้างก่อนปิด Phase 2:

1. ล็อก final layout ของแท็บ `สต็อกสินค้า` ให้แน่นก่อนแก้ script ต่อ
2. ตัดสินใจตำแหน่ง raw log/giveaway log ว่าจะอยู่แถวล่างในชีทเดิม, อยู่คอลัมน์ขวาสุด, หรือย้ายไป hidden sheet แยก
3. ทดสอบ `สร้าง LOT ใหม่` 1 รายการ แล้วเช็ก LOT_ID, ราคา, ต้นทุน, สถานะ
4. ทดสอบ `บันทึก Session` แบบไม่มีแถม/ไม่มีเสีย แล้วเช็กขายได้ = เหลือก่อน - เหลือนี้
5. ทดสอบแถมสินค้าเดียวกันและแถมข้ามสินค้า เพื่อกันการนับยอดขายผิด
6. ทดสอบราคาพิเศษและเหตุผลโปร เพื่อให้รายได้/กำไร/มาจิ้นถูกต้อง
7. หลังจาก Phase 2 นิ่งแล้ว ค่อยกลับไป Phase 3 เรื่อง analytics โปร/ราคาพิเศษ

### ⏸️ Phase 3 — โปร/ราคาพิเศษ + เหตุผล (พักไว้ก่อน)
ช่อง "ชื่อโปร/เหตุผล" (ฟอร์ม `S18` → เก็บที่ raw log `AL`) มีอยู่แล้วใน script Phase 2 แต่ยังไม่ได้ทำ analytics — Phase 3 จะเพิ่ม dropdown เหตุผลมาตรฐาน + รายงานวิเคราะห์ว่า margin หายเพราะ discount แบบไหน

### ⬜ Phase 4 — Recipe เมนูสด + หน่วยวัตถุดิบ
หน่วยย่อยใน RM (กรัม/ml/ชิ้น) + ราคาต่อหน่วยย่อย → ทำ `บันทึกสูตรอาหาร` ใช้งานจริง → ต้นทุนต่อเมนู → เมนูสดรู้ต้นทุนจริง

### ⬜ Phase 5 — งบการเงิน แยกชั้นกำไร
แยก **Gross → Contribution → Net** + แยก variable cost กับ fixed cost

### 🏁 ด่านสุดท้าย — ฐานข้อมูล + แดชบอร์ด
ปรับ `ฐานข้อมูล` ให้รองรับหลาย session ต่อ LOT **โดยไม่นับเบิกซ้ำ/ต้นทุนไม่เพี้ยน** + ต่อ Dashboard เข้ากับ KPI "ปิดยอดตรงไหม" + ranking/margin ที่ถูกต้อง

## 13. สูตรหลักที่ควรใช้หลังปรับระบบ

### ขายได้ใน session แรก

```spreadsheet
=Qty_In - Counted_Remaining - Giveaway_Qty - Waste_Qty
```

### ขายได้ใน session ถัดไป

```spreadsheet
=Prev_Remaining - Counted_Remaining - Giveaway_Qty - Waste_Qty
```

### รายได้จาก stock session

```spreadsheet
=(Normal_Sold * Normal_Price) + (Promo_Sold * Promo_Price)
```

### ต้นทุนขาย

```spreadsheet
=(Normal_Sold + Promo_Sold) * Unit_Cost
```

### ต้นทุนแถม

ต้องดึงจากสินค้า/LOT ที่ถูกแถม:

```spreadsheet
=Giveaway_Qty * Given_Product_Unit_Cost
```

### ต้นทุนเสีย

```spreadsheet
=Waste_Qty * Unit_Cost
```

### ส่วนต่างปิดวัน

```spreadsheet
=Cash_Received - Stock_Revenue
```

### สถานะปิดวัน

```spreadsheet
=IF(ABS(Difference)<=Threshold,"ปกติ","ต้องตรวจสอบ")
```

## 14. ข้อเสนอเชิงปฏิบัติสำหรับ Apps Script

ยังไม่ได้อ่าน code จริง แต่จาก workflow ควรปรับ script ตามหลักนี้

### ฟังก์ชันที่ควรมี

1. `createNewLot()`
   - อ่านวันที่/สาขา/สินค้า/จำนวนเบิกจากฟอร์ม
   - ดึงราคาขายและต้นทุนจาก master
   - สร้าง LOT_ID ใหม่
   - append เข้า `Stock Lots`

2. `appendStockSession()`
   - อ่าน LOT_ID ที่เลือก
   - ดึงเหลือล่าสุด
   - อ่านเหลือครั้งนี้, แถม, เสีย, โปร
   - คำนวณขายได้
   - append เข้า `Stock Sessions`

3. `getOpenLots()`
   - คืน LOT ที่ยังเปิดอยู่
   - ใช้เป็น dropdown ให้เลือกอัปเดต LOT เดิม

4. `closeLotIfNeeded()`
   - ถ้าเหลือ 0 หรือปิดเพราะเสีย ให้เปลี่ยนสถานะ LOT

5. `validateSession()`
   - ห้ามเหลือครั้งนี้มากกว่าเหลือครั้งก่อน ถ้าไม่มีการรับเข้าเพิ่ม
   - ห้ามขายได้ติดลบ
   - ห้ามแถมหรือเสียมากกว่ายอดที่มี

### หลักที่สำคัญ

Script ไม่ควรแก้ข้อมูลเก่าทับโดยไม่มี log

ควรใช้ append-only:

```text
กดบันทึก 1 ครั้ง = เพิ่ม 1 session row
```

ถ้าต้องแก้ไขข้อมูลย้อนหลัง ควรมี:

- edited timestamp
- edited by
- original value
- reason

แต่ถ้าระบบยังเล็ก อาจเริ่มจาก append-only ก่อน แล้วค่อยเพิ่ม audit trail ทีหลัง

## 15. มุมบริหารที่ควรคิดเพิ่ม

### 15.1 อย่าบังคับร้านให้ทำตัวเหมือนมี POS

ถ้าบังคับให้จดทุกกล่องตอนขายจริง ระบบจะละเอียดขึ้นแต่ใช้งานจริงยาก และพนักงานอาจเลิกใช้

แนวทางที่เหมาะกว่า:

- รับ workflow บ้านๆ ตามจริง
- ให้จดเงินรวม
- ให้นับของเหลือ
- ให้ระบบช่วย reconcile
- ให้แจ้งเตือนเฉพาะวันที่ยอดไม่ตรง

### 15.2 ส่วนต่างปิดวันสำคัญกว่า Dashboard สวย

ถ้ายอดเงินกับยอด stock ไม่ตรง Dashboard จะสวยแค่ไหนก็ไม่ควรเชื่อ

ดังนั้น KPI แรกที่ควรมีคือ:

```text
ปิดยอดตรงหรือไม่
```

### 15.3 Discount ต้องมีเหตุผล

ราคาโปรและราคาพิเศษควรแยกเหตุผล เพราะเป็นข้อมูลสำคัญสำหรับลดเงินจม

ถ้าวันหนึ่งเห็น margin ต่ำ จะต้องตอบได้ว่าเกิดจาก:

- ต้นทุนสูง
- ของเสีย
- แถมเยอะ
- โปรเยอะ
- ขายราคาพิเศษเยอะ
- รับเงินไม่ครบ
- นับของผิด

### 15.4 Transfer price จากบริษัทแม่ต้องชัด

Grand House รับสินค้าจาก The Grand's ถ้าต้นทุน transfer price สูงเกินไป Grand House จะดูขาดทุน แม้กลุ่มบริษัทอาจยังมีกำไร

ควรแยกวิเคราะห์:

- กำไรของ Grand House ตามราคาที่รับจากบริษัทแม่
- กำไรรวมของกลุ่ม The Grand's + Grand House

ไม่งั้นอาจตัดสินใจผิดว่าร้านลูกไม่ดี ทั้งที่ margin ถูกย้ายไปอยู่บริษัทแม่แล้ว
